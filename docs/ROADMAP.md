# 🗺️ 개발 로드맵 (Development Roadmap)

**작성일**: 2026-03-16
**상태**: 진행 전 (Not Started)
**총 예상 기간**: 9-15일

---

## 📋 로드맵 개요

이 로드맵은 Notion 독서록 웹 애플리케이션을 단계적으로 개발하기 위한 계획입니다.
각 Phase는 의존성을 고려하여 순차적으로 진행되어야 합니다.

---

## 🔄 Phase 1: 프로젝트 초기 설정 (1-2일)

### 목표
견고한 개발 기반을 구축하고 Notion API와의 연동 환경을 완성합니다.

### 작업 항목

| # | 작업 | 상세 | 예상 시간 |
|---|-----|------|---------|
| 1.1 | Next.js 프로젝트 구조 설정 | - 프로젝트 초기화 및 디렉토리 구조 생성<br/>- `src/app`, `src/components`, `src/lib` 디렉토리 구성<br/>- 기본 페이지 스켈레톤 작성 | 2시간 |
| 1.2 | Notion API 환경 구축 | - `@notionhq/client` 패키지 설치<br/>- `.env.local` 파일에 API 키 및 데이터베이스 ID 설정<br/>- 환경변수 검증 로직 구현 (`lib/config.ts`) | 1.5시간 |
| 1.3 | 기본 레이아웃 구조 생성 | - Header, Footer, Navigation 컴포넌트 구현<br/>- 기본 CSS 초기화 및 Tailwind 설정 확인<br/>- 페이지 레이아웃 템플릿 작성 | 2시간 |
| 1.4 | TypeScript 타입 정의 | - Notion API 응답 타입 정의 (`types/notion.ts`)<br/>- Book, Category, ReadStatus 등 도메인 타입 정의<br/>- API 함수 시그니처 타입 정의 | 1.5시간 |

### 완료 기준

- [ ] 프로젝트 디렉토리 구조가 올바르게 구성됨
- [ ] Notion API 키가 안전하게 저장되고 검증됨
- [ ] Header, Footer 컴포넌트가 모든 페이지에 표시됨
- [ ] 기본 레이아웃이 반응형으로 동작함
- [ ] TypeScript 타입 에러 없음
- [ ] `npm run dev` 실행 성공 및 브라우저에서 기본 페이지 표시

### 이 Phase가 중요한 이유

견고한 기반 없이는 기능 개발이 어렵고, 나중에 구조를 변경하면 모든 컴포넌트를 수정해야 합니다.

## 🔄 Phase 2: 공통 모듈 개발 (2-3일)

### 목표
모든 기능에서 재사용될 공통 모듈, 컴포넌트, 타입을 개발합니다.

### 작업 항목

| # | 작업 | 상세 | 예상 시간 |
|---|-----|------|---------|
| 2.1 | Notion API 공통 함수 | - `fetchPages()`: 데이터베이스의 모든 책 조회<br/>- `fetchPageContent()`: 특정 책의 상세 내용 조회<br/>- 에러 처리 및 재시도 로직 구현<br/>- 캐싱 전략 검토 | 2.5시간 |
| 2.2 | 공통 UI 컴포넌트 | - Header: 네비게이션 메뉴 포함<br/>- Footer: 사이트 정보<br/>- BookCard: 책 정보 카드 (목록 페이지용)<br/>- Badge: 카테고리, 태그, 상태 표시 | 2시간 |
| 2.3 | 공통 타입 정의 | - `Book`: 책 데이터 타입<br/>- `Category`: 카테고리 타입<br/>- `ReadStatus`: 읽기 상태 타입<br/>- `NotionBlockResponse`: Notion 블록 응답 타입 | 1.5시간 |
| 2.4 | 유틸리티 함수 | - `formatDate()`: 날짜 포맷팅<br/>- `extractBookId()`: Notion 페이지 ID에서 책 ID 추출<br/>- `parseNotionContent()`: Notion 콘텐츠 파싱 | 1.5시간 |

### 완료 기준

- [ ] `lib/notion.ts`에 모든 Notion API 함수 구현
- [ ] `src/components/common/` 디렉토리에 공통 컴포넌트 완성
- [ ] `types/index.ts`에 모든 도메인 타입 정의
- [ ] `lib/utils.ts`에 유틸리티 함수 구현
- [ ] 모든 함수에 JSDoc 주석 작성
- [ ] `npm run check-all` 통과

### 이 Phase가 중요한 이유

모든 기능에서 재사용되는 코드를 먼저 만들어야 중복을 방지하고,
나중에 유지보수가 쉬워집니다. 한 번 작성하면 여러 페이지에서 사용할 수 있습니다.

## 🔄 Phase 3: 핵심 기능 개발 (3-4일)

### 목표
블로그의 기본이 되는 글 목록 및 상세 페이지를 완성합니다.

### 작업 항목

| # | 작업 | 상세 | 예상 시간 |
|---|-----|------|---------|
| 3.1 | 홈 페이지 (`/`) | - 최근 책 목록 (상위 5개) 표시<br/>- 현재 읽는 책 목록<br/>- 이달의 독서 통계 카드<br/>- 빠른 접근 링크 | 2시간 |
| 3.2 | 책 상세 페이지 (`/book/[id]`) | - Notion 페이지 정보 표시<br/>- 책 제목, 카테고리, 태그, 출판일<br/>- 읽기 상태 및 읽기 시작/완료 날짜<br/>- Notion 콘텐츠(감상문) 렌더링 | 2.5시간 |
| 3.3 | 카테고리별 목록 (`/category/[categoryName]`) | - 해당 카테고리의 모든 책 목록<br/>- 책별 읽기 상태 배지<br/>- 필터링 옵션 (상태별, 날짜순)<br/>- 페이지네이션 (선택사항) | 2시간 |
| 3.4 | 동적 라우팅 구현 | - Next.js 동적 라우팅 설정<br/>- `generateStaticParams()` 또는 `generateMetadata()`<br/>- 404 페이지 처리 | 1.5시간 |

### 완료 기준

- [ ] `/` 페이지에서 최근 책 목록과 통계 표시
- [ ] `/book/[id]` 페이지에서 상세 정보 및 감상문 표시
- [ ] `/category/[categoryName]` 페이지에서 카테고리별 책 목록 표시
- [ ] 모든 페이지에서 로딩 상태 표시
- [ ] 에러 발생 시 에러 페이지 표시
- [ ] 모든 링크가 정상적으로 동작
- [ ] `npm run build` 성공

### 이 Phase가 중요한 이유

블로그의 가장 기본이 되는 기능이며, 사용자가 실제로 콘텐츠를 볼 수 있는 단계입니다.
이 Phase가 완료되면 MVP의 80%가 완성됩니다.

## 🔄 Phase 4: 추가 기능 개발 (2-3일)

### 목표
사용성을 높이는 추가 기능들을 구현합니다.

### 작업 항목

| # | 작업 | 상세 | 예상 시간 |
|---|-----|------|---------|
| 4.1 | 카테고리 필터링 | - 홈 페이지에서 카테고리별 필터<br/>- 목록 페이지에서 다중 선택 필터<br/>- 필터 상태 URL에 반영 (`?category=소설&status=읽기완료`) | 1.5시간 |
| 4.2 | 검색 기능 | - 전역 검색 입력 필드 (Header)<br/>- 책 제목, 태그로 검색<br/>- 검색 결과 페이지 (`/search?q=...`) | 1.5시간 |
| 4.3 | 캘린더 뷰 (`/calendar`) | - 월별 캘린더 표시<br/>- 읽기 시작/완료 날짜에 책 표시<br/>- 상태별 색상 구분<br/>- 주간/월간 뷰 토글 (선택사항) | 2.5시간 |
| 4.4 | 통계 페이지 (`/stats`) | - 주간 독서 현황<br/>- 월간 독서 현황<br/>- 카테고리별 독서 분포 차트<br/>- 읽기 상태별 책 수 | 2시간 |
| 4.5 | SEO 최적화 | - 페이지별 `metadata` 설정 (메타데이터)<br/>- Open Graph 태그<br/>- Sitemap 생성<br/>- robots.txt 설정 | 1.5시간 |

### 완료 기준

- [ ] 필터링이 정상적으로 동작하고 URL에 반영됨
- [ ] 검색 결과가 정확하게 표시됨
- [ ] 캘린더가 모든 책 날짜를 올바르게 표시
- [ ] 통계 페이지의 차트가 정확한 데이터 표시
- [ ] 모든 페이지의 메타데이터 설정 완료
- [ ] Google Search Console에 sitemap 등록 가능 상태

### 이 Phase가 중요한 이유

핵심 기능이 완성된 후 부가 기능을 추가하여 사용성을 높입니다.
이 단계에서는 사용자 경험을 개선하는 데 중점을 둡니다.

## 🔄 Phase 5: 최적화 및 배포 (1-2일)

### 목표
성능을 최적화하고 배포 가능한 상태로 만듭니다.

### 작업 항목

| # | 작업 | 상세 | 예상 시간 |
|---|-----|------|---------|

| 5.2 | 반응형 디자인 개선 | - 모바일 화면 테스트<br/>- 태블릿 화면 테스트<br/>- 데스크톱 화면 테스트<br/>- 다크모드 지원 (선택사항) | 1.5시간 |
| 5.3 | 최종 검사 및 테스트 | - `npm run check-all` 실행 (모든 검사 통과)<br/>- `npm run build` 실행 (빌드 성공)<br/>- 모든 페이지 수동 테스트<br/>- 에러 핸들링 검증 | 1시간 |

### 완료 기준

- [ ] Lighthouse 점수: Performance ≥ 90, Accessibility ≥ 95
- [ ] 모든 페이지가 반응형으로 동작
- [ ] `npm run check-all` 100% 통과
- [ ] `npm run build` 성공 (경고 없음)
- [ ] Vercel 또는 선택된 플랫폼에 배포 완료
- [ ] 배포된 URL에서 모든 기능 정상 동작 확인

### 이 Phase가 중요한 이유

기능이 완성된 후 품질을 향상시킵니다.
사용자가 빠른 속도와 좋은 경험을 얻도록 최적화하는 단계입니다.
